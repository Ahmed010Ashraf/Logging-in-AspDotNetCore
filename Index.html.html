LESSON 1 · FUNDAMENTALS
How .NET Handles Logging
Understand the complete architecture behind ASP.NET Core's logging system — from the abstraction layer to providers and categories.

🏗️ The Big Picture

ASP.NET Core logging is built on a clean, provider-based architecture. Microsoft designed it so your application code never depends on a specific logging library — you always code against an abstraction, and you can swap the actual logging "sink" (Console, File, Database, Cloud…) without changing a single line of your application code.

ASP.NET Core Logging Architecture

Your Code ↓ uses
ILogger<T> ↓ abstraction defined in
Microsoft.Extensions.Logging ↓ sends to
ILoggerFactory ↓ dispatches to
Provider 1 (Console)   Provider 2 (File)   Provider 3 (Serilog)

This pattern is called the Provider Pattern. Your code logs once, and all registered providers receive the message.

📦 The Three Core Packages

Abstractions
Microsoft.Extensions.Logging.Abstractions

Contains the interfaces: ILogger, ILogger<T>, ILoggerFactory, ILoggerProvider. This is what your code references.

Core Implementation
Microsoft.Extensions.Logging

Implements the factory, DI registration, log filtering, and the logger itself. Included in ASP.NET Core by default.

Console Provider
Microsoft.Extensions.Logging.Console

The built-in provider that writes to stdout. Automatically registered in web apps via WebApplication.CreateBuilder().

Configuration
Microsoft.Extensions.Configuration

Lets you control log levels per category from appsettings.json or environment variables — no code recompile needed.

🔑 Key Concepts You Must Know

1. Log Category

Every log entry has a category — this is simply the fully-qualified name of the class that created the logger. When you inject ILogger<MyService>, the category becomes "YourApp.Services.MyService". This lets you filter logs with surgical precision — you can set one noise level for Microsoft's own framework logs and a different level for your own code.

2. Log Event ID

Optionally, each log call can include an EventId — an integer + name pair. This is extremely useful in large systems to search, alert, or route specific events. Example: event 5001 = "Order placed", event 5002 = "Payment failed".

3. Log Provider

A provider is the thing that decides where your log goes — Console, Debug, EventLog, Application Insights, a file, a database, etc. You register providers during app startup. Multiple providers can be active at the same time.

4. ILoggerFactory vs ILogger<T>

ILoggerFactory

The central hub. Creates loggers, holds all providers, applies filters. You rarely use this directly in your own code.

ILogger<T>

What you inject into your classes. Pre-scoped to the class name as category. This is what you use 99% of the time.

🚀 Your First Logger — Hello Logging

In a brand new ASP.NET Core Web API, logging is already wired up. Here's how minimal it is to start:

C# — Program.cs (Minimal API)
var builder = WebApplication.CreateBuilder(args);

// ✅ Logging is already registered by CreateBuilder()!
// Console, Debug, EventSource providers are included automatically.

var app = builder.Build();

app.MapGet("/hello", (ILogger<Program> logger) =>
{
    logger.LogInformation("Hello endpoint was called!");
    return "Hello World";
});

app.Run();
💡

WebApplication.CreateBuilder() automatically calls logging.AddConsole(), logging.AddDebug(), and reads your appsettings.json Logging section. You get a fully working logging system with zero configuration.

📊 What Happens When You Call logger.LogInformation()?

1

Log entry is created

A log entry object is built containing: category name, log level, event ID, message, exception (if any), and timestamp.

2

Filters are checked

The framework checks: "Is this log level enabled for this category?" If the minimum level is Warning and you log Information, it's dropped here — no allocations, no work.

3

Each provider is called

For each registered provider (Console, File, etc.), the framework checks the provider-specific filter, then calls provider.Log() if it passes.

4

Provider formats and outputs

Each provider formats the entry in its own way and writes it to its destination — console, file, cloud service, etc.

🧠 Quick Check

In ASP.NET Core, when you call logger.LogInformation(), which interface are you calling it on?
✅ Correct! ILogger<T> is what you inject and use in your classes. ILoggerFactory creates loggers, and ILoggerProvider is implemented by each sink.

Log Levels Explained
Master all 6 log levels — understand what each means, when to use it, and how they control verbosity across environments.

📊 The 6 Log Levels (Lowest → Highest)

LevelValueBadgeUse when…
Trace0TRACEExtremely detailed debugging — function entry/exit, variable values. Never in production.
Debug1DEBUGUseful during development — query parameters, method arguments, flow tracking.
Information2INFONormal application flow milestones — user logged in, order created, service started.
Warning3WARNSomething unexpected but recoverable — retry attempted, deprecated API used, config missing.
Error4ERRORA failure that affects a specific request/operation — exception caught, DB query failed.
Critical5CRITICALSystem-wide failure — out of memory, disk full, complete service outage. Wake someone up!
💡

There is also LogLevel.None (6) — a special value used in configuration only to completely disable all logging for a category. You never log at level None.

✍️ How to Write Each Level

C# — All log level methods
public class OrderService
{
    private readonly ILogger<OrderService> _logger;

    public OrderService(ILogger<OrderService> logger)
    {
        _logger = logger;
    }

    public void ProcessOrder(int orderId, string userId)
    {
        // Trace: ultra-detailed, only when debugging deep issues
        _logger.LogTrace("Entering ProcessOrder. orderId={OrderId}, userId={UserId}", orderId, userId);

        // Debug: helpful in development
        _logger.LogDebug("Looking up order {OrderId} from database", orderId);

        // Information: normal happy path
        _logger.LogInformation("Processing order {OrderId} for user {UserId}", orderId, userId);

        // Warning: something off but not broken
        _logger.LogWarning("Order {OrderId} is using a deprecated payment method", orderId);

        // Error: something failed (pass the exception!)
        try { /* ... */ }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Failed to process order {OrderId}", orderId);
        }

        // Critical: system is going down
        _logger.LogCritical("Payment provider is completely unreachable. System halted.");
    }
}
⚠️

Always pass the exception as the first argument to LogError/LogCritical — not inside the string. LogError(ex, "message") ✅ vs LogError("Error: " + ex.Message) ❌. The first form captures the full stack trace properly.

🌍 Log Levels by Environment

Development

Set minimum to Debug or even Trace. You want maximum detail to understand what's happening.

Staging

Set to Information. Enough to verify business logic without drowning in noise.

Production

Set to Warning or Information. Noise costs money (log storage) and makes real issues harder to find.

Microsoft Framework Logs

Set Microsoft.* to Warning even in dev — framework logs are extremely noisy at lower levels.

⚡ IsEnabled — Performance Tip

Building complex log messages (string concatenation, object serialization) has a cost — even if the log is filtered out. Use IsEnabled() to guard expensive operations:

C# — IsEnabled guard
// ❌ BAD — serializes the entire order object even if Debug is disabled
_logger.LogDebug("Full order: {Order}", JsonSerializer.Serialize(order));

// ✅ GOOD — only pays the cost if Debug level is actually enabled
if (_logger.IsEnabled(LogLevel.Debug))
{
    _logger.LogDebug("Full order: {Order}", JsonSerializer.Serialize(order));
}
💡

For simple string interpolation, ASP.NET Core already skips allocation when the level is disabled. The guard above is specifically needed when creating the arguments themselves is expensive (serialization, DB calls, etc.).

🧠 Quick Check

Your application catches an exception in a payment processing method. The payment failed for that specific request but the system is still running. Which log level should you use?
✅ Correct! Error is right for a failure that affects a specific operation. Critical would be if the entire payment system was down and no one could pay.
LESSON 3 · FUNDAMENTALS
ILogger & Dependency Injection
Learn all the ways to get a logger into your classes, and understand the powerful message template syntax.

💉 Getting ILogger via DI

The standard way — inject ILogger<T> into your constructor. The type parameter T becomes the log category (the class name).

C# — Constructor Injection (most common)
public class UserService
{
    private readonly ILogger<UserService> _logger;
    private readonly AppDbContext _db;

    public UserService(ILogger<UserService> logger, AppDbContext db)
    {
        _logger = logger; // Category: "YourApp.Services.UserService"
        _db = db;
    }

    public async Task<User?> GetUserAsync(int id)
    {
        _logger.LogDebug("Fetching user {UserId}", id);
        var user = await _db.Users.FindAsync(id);

        if (user == null)
            _logger.LogWarning("User {UserId} not found", id);

        return user;
    }
}
C# — In Controllers (same pattern)
[ApiController]
[Route("api/[controller]")]
public class OrdersController : ControllerBase
{
    private readonly ILogger<OrdersController> _logger;

    public OrdersController(ILogger<OrdersController> logger)
    {
        _logger = logger;
    }

    [HttpGet("{id}")]
    public IActionResult GetOrder(int id)
    {
        _logger.LogInformation("Getting order {OrderId}", id);
        // ...
        return Ok();
    }
}

📝 Message Templates — The Right Way to Log

ASP.NET Core uses message templates, not string interpolation. The placeholders {Name} in curly braces are named, and the values are passed separately.

C# — Message Templates
// ✅ CORRECT — use message templates
_logger.LogInformation("User {UserId} placed order {OrderId} for ${Amount}",
    userId, orderId, amount);

// ❌ WRONG — string interpolation (defeats structured logging)
_logger.LogInformation($"User {userId} placed order {orderId} for ${amount}");

// ❌ WRONG — string concatenation (same problem)
_logger.LogInformation("User " + userId + " placed order " + orderId);
⚠️

With string interpolation ($"..."), the message is already a plain string — the individual values are lost. With message templates, structured logging providers (like Serilog + Seq) can index and query by UserId, OrderId, etc. separately. This is one of the most important habits in .NET logging.

Destructuring with @

Use @ prefix to serialize an entire object as structured data:

C# — Object Destructuring
var order = new { Id = 42, Customer = "Ahmed", Total = 199.99 };

// Without @ — logs order.ToString()
_logger.LogInformation("Order created: {Order}", order);

// With @ — serializes all properties as structured data
_logger.LogInformation("Order created: {@Order}", order);
// Output: Order created: { Id: 42, Customer: "Ahmed", Total: 199.99 }

🏷️ EventId — Tagging Your Log Entries

C# — Using EventId
// Define your event IDs as constants
public static class LogEvents
{
    public static readonly EventId UserLoggedIn  = new(1000, "UserLoggedIn");
    public static readonly EventId OrderPlaced   = new(2000, "OrderPlaced");
    public static readonly EventId PaymentFailed = new(3001, "PaymentFailed");
}

// Use them when logging
_logger.LogInformation(LogEvents.OrderPlaced,
    "Order {OrderId} placed by {UserId}", orderId, userId);

_logger.LogError(LogEvents.PaymentFailed, exception,
    "Payment failed for order {OrderId}", orderId);
💡

EventIds are optional but highly recommended in production systems. They let you search "all PaymentFailed events in the last hour" in your logging platform, set up alerts on specific events, and create dashboards per event type.

🔧 LoggerMessage.Define — High-Performance Logging

For hot paths (high-frequency code), use LoggerMessage.Define to pre-compile log actions. It avoids boxing and is 3-10x faster than regular LogXxx calls:

C# — High-Performance Logging
public partial class OrderService { private readonly ILogger<OrderService> _logger; // Pre-compiled log action — zero allocation on hot path [LoggerMessage( EventId = 2000, Level = LogLevel.Information, Message = "Order {OrderId} placed by user {UserId}")] private partial void LogOrderPlaced(int orderId, string userId); public void PlaceOrder(int orderId, string userId) { LogOrderPlaced(orderId, userId); // Compiler generates the fast implementation } }
💡

The [LoggerMessage] source generator (available since .NET 6) automatically generates the high-performance logging code at compile time. Use it for any log call that happens in a tight loop or per-request critical path.

🧠 Quick Check

Why should you use message templates ("User {Id} logged in") instead of string interpolation ($"User {id} logged in")?
✅ Correct! Message templates keep the values as separate named properties. Structured logging tools can then index, filter, and query on UserId = 42 specifically. Interpolation bakes everything into one flat string — all structure is lost.

appsettings.json Configuration
Master the Logging configuration section — control levels per category, per provider, and per environment without touching code.

📄 The Full Configuration Structure

JSON — appsettings.json (Full Example)
{
  "Logging": {
    "LogLevel": {
      // Default minimum level for ALL categories and ALL providers
      "Default": "Information",

      // Microsoft framework internals — keep these quiet!
      "Microsoft": "Warning",

      // ASP.NET Core hosting — you usually don't need its Trace/Debug noise
      "Microsoft.AspNetCore": "Warning",

      // Entity Framework — SQL queries at Debug, warnings at Warning
      "Microsoft.EntityFrameworkCore.Database.Command": "Warning",

      // YOUR application — full detail
      "MyApp": "Debug",

      // A specific service — even more detail
      "MyApp.Services.PaymentService": "Trace"
    },
    "Console": {
      // Provider-specific overrides (Console provider only)
      "LogLevel": {
        "Default": "Warning"
      },
      "FormatterName": "json"
    }
  }
}

🌿 Environment-Specific Config Files

ASP.NET Core loads configuration in this order — later files override earlier ones:

1

appsettings.json

Base config — applies everywhere. Put safe defaults here.

2

appsettings.Development.json

Overrides for local development. Set verbose levels here.

3

appsettings.Production.json

Overrides for production. Set minimal levels here.

4

Environment Variables

Highest priority. Use Logging__LogLevel__Default=Warning in containers/cloud.

JSON — appsettings.Development.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Debug",
      "Microsoft.AspNetCore": "Information",
      "MyApp": "Trace"
    }
  }
}
JSON — appsettings.Production.json
{
  "Logging": {
    "LogLevel": {
      "Default": "Warning",
      "Microsoft": "Warning",
      "MyApp": "Information"
    }
  }
}

🔑 Category Matching Rules

Categories use a longest prefix match rule. Given the category MyApp.Services.OrderService, the framework checks:

Matching Priority (highest to lowest)
1. "MyApp.Services.OrderService"  ← exact match (highest priority)
2. "MyApp.Services"               ← prefix match
3. "MyApp"                        ← prefix match
4. "Default"                      ← fallback (lowest priority)
💡

You can silence the extremely noisy EF Core SQL logging with: "Microsoft.EntityFrameworkCore": "Warning" while keeping your own app's logs verbose. This is one of the most practical config tricks in real projects.

⚙️ Configuring Logging in Program.cs

C# — Program.cs — Full Logging Setup
var builder = WebApplication.CreateBuilder(args);

builder.Logging
    // Remove all default providers
    .ClearProviders()

    // Add only the providers you want
    .AddConsole()
    .AddDebug()

    // Set minimum level programmatically (config file takes precedence)
    .SetMinimumLevel(LogLevel.Information)

    // Fine-grained filter via code
    .AddFilter("Microsoft", LogLevel.Warning)
    .AddFilter("MyApp", LogLevel.Debug)

    // Read config from appsettings.json (this is automatic, shown for clarity)
    .AddConfiguration(builder.Configuration.GetSection("Logging"));
⚠️

ClearProviders() removes Console, Debug, and EventSource that are auto-registered. Call it when you want full control — e.g., when adding Serilog, you don't want duplicate output.

🧠 Quick Check

You have a category MyApp.Services.EmailService and your config has "MyApp": "Warning" and "MyApp.Services": "Debug". Which level applies?
✅ Correct! Longest prefix match wins. MyApp.Services is a longer (more specific) match for MyApp.Services.EmailService than MyApp, so Debug applies.
LESSON 5 · CONFIGURATION
Structured Logging & Scopes
Learn structured logging (logs as data, not text) and how log scopes let you attach context properties to a group of related log entries.

📊 Structured Logging vs Plain Text

❌ Plain Text Logging

"User 42 placed order 999 at 2024-01-15"

Just a string. You can grep it, but you can't query all orders over $100 by user 42 without regex hacks.

✅ Structured Logging

{ UserId: 42, OrderId: 999, Amount: 199.99, Timestamp: "..." }

A log is a record with typed fields. Query WHERE UserId=42 AND Amount > 100.

💡

When you use message templates ("User {UserId} placed order {OrderId}"), the logging framework stores UserId and OrderId as separate named properties — not just text. Providers like Serilog + Seq, Application Insights, or Elasticsearch can then index and query these as structured data.

🔭 Log Scopes — Context That Follows You

A scope is a block of ambient context attached to all log entries made within it. Classic use case: attach the current HTTP request ID or user ID to every log in that request, without passing it explicitly to every method.

C# — Using Log Scopes
public async Task ProcessOrderAsync(int orderId, string userId)
{
    // Everything logged inside this using block gets OrderId and UserId attached
    using (var scope = _logger.BeginScope(
        new Dictionary<string, object>
        {
            ["OrderId"] = orderId,
            ["UserId"] = userId,
            ["CorrelationId"] = Guid.NewGuid()
        }))
    {
        _logger.LogInformation("Starting order processing");
        await ValidateOrderAsync(orderId);    // Logs here also get the scope
        await ChargePaymentAsync(orderId);   // And here
        _logger.LogInformation("Order processing complete");
    }
    // Scope is disposed — context removed from subsequent logs
}

All three log entries above will have OrderId, UserId, and CorrelationId in their structured properties — even though you didn't explicitly include them in each log call.

🔗 Correlation IDs — Tracing a Request End-to-End

A correlation ID is a unique identifier you attach to every log entry for a given HTTP request. This lets you find all logs for a single user request — even across microservices.

C# — Correlation ID Middleware
public class CorrelationIdMiddleware
{
    private readonly RequestDelegate _next;
    private readonly ILogger<CorrelationIdMiddleware> _logger;

    public CorrelationIdMiddleware(RequestDelegate next,
        ILogger<CorrelationIdMiddleware> logger)
    {
        _next = next;
        _logger = logger;
    }

    public async Task InvokeAsync(HttpContext context)
    {
        // Get from header (for microservices) or generate new
        var correlationId = context.Request.Headers["X-Correlation-ID"]
            .FirstOrDefault() ?? Guid.NewGuid().ToString();

        context.Response.Headers.TryAdd("X-Correlation-ID", correlationId);

        // Attach to ALL logs for this request
        using (_logger.BeginScope(new { CorrelationId = correlationId }))
        {
            await _next(context);
        }
    }
}

// Register in Program.cs
app.UseMiddleware<CorrelationIdMiddleware>();

⚙️ Enable Scopes in Configuration

Scopes must be enabled in the console provider config (they're on by default in most providers, but good to be explicit):

JSON — appsettings.json
{
  "Logging": {
    "Console": {
      "IncludeScopes": true,
      "FormatterName": "json"
    }
  }
}

With FormatterName: "json", the console outputs each log entry as a full JSON object — great for log aggregation tools like Datadog, Elastic, or Splunk that ingest JSON lines.

🧠 Quick Check

You call _logger.BeginScope(new { RequestId = "abc-123" }). What happens to log entries written inside the using block?
✅ Correct! Every log entry made inside the using block — including in called methods — automatically inherits the scope's properties. This is incredibly powerful for attaching request context to all logs.

🖥️ Console Provider — Formatters

The Console provider has three built-in formatters you can configure:

C# — Console Formatter Options
builder.Logging.AddConsole(options =>
{

builder.Logging.AddConsole()

Human-readable colored output. Best for local development.

json

Machine-readable JSON lines. Best for production containers and log aggregators.

systemd

Compatible with systemd journal format. For Linux systemd-managed services.

Custom

Implement ConsoleFormatter to fully control the output format.

LESSON 7 · ADVANCED
Serilog Integration
Replace the built-in logging pipeline with Serilog — the most popular structured logging library for .NET — and wire it to powerful sinks.

🤔 Why Serilog?

Built-in Logging

Good defaults. Limited sinks. Console formatter is basic. No file rolling. Hard to write to multiple destinations with different formats.

Serilog

80+ sinks (File, SQL, Elastic, Seq, Datadog…). Enrichers. Output templates. Destructuring policies. Rolling files. Async sink. Industry standard.

📦 Installation

Shell — NuGet Packages
dotnet add package Serilog.AspNetCore
dotnet add package Serilog.Sinks.Console
dotnet add package Serilog.Sinks.File
dotnet add package Serilog.Enrichers.Environment
dotnet add package Serilog.Enrichers.Thread
dotnet add package Serilog.Settings.Configuration

⚙️ Serilog Setup in Program.cs

C# — Program.cs with Serilog
using Serilog;
using Serilog.Events;

// Bootstrap logger — captures startup errors before DI is ready
Log.Logger = new LoggerConfiguration()
    .MinimumLevel.Override("Microsoft", LogEventLevel.Warning)
    .WriteTo.Console()
    .CreateBootstrapLogger();

try
{
    Log.Information("Starting application");
    var builder = WebApplication.CreateBuilder(args);

    // Replace built-in logging with Serilog
    builder.Host.UseSerilog((context, services, configuration) =>
        configuration
            // Read from appsettings.json "Serilog" section
            .ReadFrom.Configuration(context.Configuration)
            .ReadFrom.Services(services)

            // Minimum levels
            .MinimumLevel.Information()
            .MinimumLevel.Override("Microsoft", LogEventLevel.Warning)
            .MinimumLevel.Override("Microsoft.EntityFrameworkCore", LogEventLevel.Warning)

            // Enrichers — add machine name, thread ID, etc. to every log
            .Enrich.FromLogContext()
            .Enrich.WithMachineName()
            .Enrich.WithThreadId()
            .Enrich.WithEnvironmentName()

            // Sinks
            .WriteTo.Console(new ExpressionTemplate(
                "[{@t:HH:mm:ss} {@l:u3}] [{SourceContext}] {@m}\n{@x}"))
            .WriteTo.File(
                path: "logs/app-.log",
                rollingInterval: RollingInterval.Day,
                retainedFileCountLimit: 30,
                outputTemplate: "{Timestamp:yyyy-MM-dd HH:mm:ss} [{Level:u3}] {Message:lj}{NewLine}{Exception}"
            )
    );

    var app = builder.Build();
    app.UseSerilogRequestLogging(); // Auto-log every HTTP request

    app.Run();
}
catch (Exception ex)
{
    Log.Fatal(ex, "Application terminated unexpectedly");
}
finally
{
    await Log.CloseAndFlushAsync(); // Flush buffers on shutdown
}

🧠 Quick Check

When using Serilog, why should you call Log.CloseAndFlushAsync() in the finally block?
✅ Correct! Serilog (and many other sinks) buffer log entries asynchronously for performance. If the process exits without flushing, the last batch of logs can be lost. CloseAndFlushAsync() ensures they're all written.
LESSON 8 · ADVANCED
Custom Log Provider
Build your own logging provider from scratch — understand the three interfaces you need and how the pipeline fits together.

🧩 The Three Interfaces

To create a custom provider you implement three interfaces:

ILoggerProvider

The factory. Creates ILogger instances for each category. Registered with DI. Implements IDisposable.

ILogger

The actual logger for a single category. Has Log(), IsEnabled(), and BeginScope().

IExternalScopeProvider

Optional but recommended. Lets your provider access scope state (the properties set by BeginScope()).

💻 Complete Custom Provider Example

Let's build a provider that logs to an in-memory collection (useful for testing):

C# — Custom In-Memory Logger
// 1. The log entry record
public record LogEntry(
    LogLevel Level,
    string Category,
    EventId EventId,
    string Message,
    Exception? Exception,
    DateTimeOffset Timestamp);

// 2. The ILogger implementation
public class InMemoryLogger : ILogger
{
    private readonly string _category;
    private readonly ConcurrentQueue<LogEntry> _entries;
    private IExternalScopeProvider? _scopeProvider;

    public InMemoryLogger(string category, ConcurrentQueue<LogEntry> entries)
    {
        _category = category;
        _entries = entries;
    }

    public void SetScopeProvider(IExternalScopeProvider provider)
        => _scopeProvider = provider;

    public IDisposable? BeginScope<TState>(TState state) where TState : notnull
        => _scopeProvider?.Push(state) ?? NullScope.Instance;

    public bool IsEnabled(LogLevel logLevel) => logLevel != LogLevel.None;

    public void Log<TState>(
        LogLevel logLevel, EventId eventId, TState state,
        Exception? exception, Func<TState, Exception?, string> formatter)
    {
        if (!IsEnabled(logLevel)) return;

        _entries.Enqueue(new LogEntry(
            logLevel, _category, eventId,
            formatter(state, exception),
            exception,
            DateTimeOffset.UtcNow));
    }
}

// 3. The ILoggerProvider
[ProviderAlias("InMemory")]
public class InMemoryLoggerProvider : ILoggerProvider, ISupportExternalScope
{
    private readonly ConcurrentQueue<LogEntry> _entries = new();
    private readonly ConcurrentDictionary<string, InMemoryLogger> _loggers = new();
    private IExternalScopeProvider _scopeProvider = new LoggerExternalScopeProvider();

    public IReadOnlyCollection<LogEntry> Entries => _entries.ToArray();

    public ILogger CreateLogger(string categoryName)
    {
        return _loggers.GetOrAdd(categoryName, name =>
        {
            var logger = new InMemoryLogger(name, _entries);
            logger.SetScopeProvider(_scopeProvider);
            return logger;
        });
    }

    public void SetScopeProvider(IExternalScopeProvider scopeProvider)
    {
        _scopeProvider = scopeProvider;
        foreach (var logger in _loggers.Values)
            logger.SetScopeProvider(scopeProvider);
    }

    public void Dispose() { }
}

// 4. Extension method for easy registration
public static class InMemoryLoggerExtensions
{
    public static ILoggingBuilder AddInMemory(this ILoggingBuilder builder)
    {
        builder.Services.AddSingleton<InMemoryLoggerProvider>();
        builder.Services.AddSingleton<ILoggerProvider>(
            sp => sp.GetRequiredService<InMemoryLoggerProvider>());
        return builder;
    }
}

// 5. Register and use
builder.Logging.AddInMemory();

// In tests — read all logged entries:
var provider = app.Services.GetRequiredService<InMemoryLoggerProvider>();
var errors = provider.Entries.Where(e => e.Level >= LogLevel.Error);

🧠 Quick Check

In a custom logging provider, what is the role of ILoggerProvider vs ILogger?
✅ Correct! ILoggerProvider is a factory — the framework calls CreateLogger(categoryName) on it for each unique category. Each resulting ILogger instance handles logging for that specific category (class).

Production Best Practices
Everything you need to run reliable, observable, and efficient logging in real production systems.

✅ The Production Logging Checklist

1

Use structured logging — always use message templates

Never use string interpolation in log calls. This single habit is the foundation of queryable, actionable logs.

2

Set Microsoft.* to Warning in production

Framework logs at Debug/Info are extremely noisy and expensive. Silence them unless you're debugging a specific framework issue.

3

Use log scopes for correlation IDs

Every HTTP request should carry a correlation ID through all log entries. Implement a middleware that creates the scope.

4

Never log sensitive data

Passwords, credit card numbers, tokens, PII — never in logs. Use redaction or masking. Consider a [SensitiveData] attribute pattern.

5

Always pass exceptions to LogError/LogCritical

LogError(ex, "message") — the exception is the first parameter, not inside the string. This captures the full stack trace.

6

Use async/buffered sinks in production

Synchronous logging on the hot path kills throughput. Serilog's Async wrapper or file sinks are buffered by default.

7

Define EventIds for important business events

EventIds let you alert on specific events, build dashboards, and search logs with precision.

8

Log at the right level

Think before choosing a level. Over-warning is as bad as over-debugging. If everything is a warning, nothing is.
